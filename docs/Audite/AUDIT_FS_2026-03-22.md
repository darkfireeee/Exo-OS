# Audit approfondi du module `fs` (Exo-OS)

Date: 2026-03-22
Périmètre: `kernel/src/fs/**`
Objectif: base de refonte/correction du sous-système fichiers

---

## 1) Résumé exécutif

Le module `fs` est la couche 3 du noyau Exo-OS.
Il encapsule ExoFS, son dispatcher syscall dédié, ses pipelines stockage, snapshot, relation, quota, recovery et observabilité.
Le périmètre est massif (289 fichiers source).

Points forts:
- découpage modulaire extrêmement fin.
- API syscall centralisée (500 → 520).
- séparation claire des sous-domaines (storage/gc/snapshot/relation/quota/etc.).
- nombreuses protections concurrence (SpinLock, atomiques, locks dédiés).

Points de vigilance:
- dette technique documentée (stubs/placeholder/TODO dans certains sous-modules).
- couplage fort entre epoch commit, writeback, GC et recovery.
- surface API large: forte exigence de non-régression.
- volume très élevé de code => besoin de priorisation par lots.

---

## 2) Position dans l’architecture noyau

`fs` dépend de:
- `memory`
- `scheduler`
- `process`
- `security`

`fs` expose:
- montage/init ExoFS
- enregistrement VFS bridge
- dispatch syscall ExoFS
- compat POSIX partielle (bridge)

Invariants globaux:
- init unique (`EXOFS_INITIALIZED`).
- recovery avant exposition service.
- commit epoch coordonné (verrou unique).
- arrêt propre via shutdown commit.

---

## 3) Fonctionnement de haut niveau

Séquence type au boot:
1. `exofs_init(disk_size_bytes)`
2. `boot_recovery_sequence(...)`
3. init bridge POSIX
4. spawn kthread GC bas-prio
5. enregistrement VFS

Séquence type en shutdown:
1. vérification init
2. commit epoch forcé
3. tolérance `CommitInProgress`
4. `EXOFS_INITIALIZED=false`

Chemins critiques:
- chemin écriture objet
- commit epoch
- writeback/flush
- snapshots
- fsck/recovery

## 8) Concurrence: Mutex / SpinLock / atomiques

Patterns majeurs:
- `SpinLock` (scheduler sync) omniprésent dans GC, storage, snapshot, epoch pin/commit.
- spinlock custom `AtomicU64 + UnsafeCell` dans plusieurs sous-systèmes (queues/rings/tables).
- `AtomicBool` pour état global init.
- `AtomicU64` pour métriques/compteurs.

Chemins à haut risque de contention:
- commit epoch lock global
- tables FD objet
- caches metadata/object/path
- registres GC globaux

Règles de refonte:
- éviter I/O sous lock long.
- documenter lock ordering inter-modules.
- privilégier sections critiques courtes.

---

## 9) Usages `&str` significatifs

`&str` présent dans:
- noms/labels d’opérations et catégories (`name() -> &'static str`)
- messages observabilité/audit
- composants path (conversion UTF-8 validée)
- helpers export/reporting

Enjeux:
- éviter allocations inutiles sur hot path.
- préserver frontières UTF-8 validées.
- distinguer `&'static str` diagnostics constants vs `&str` runtime.

## 13) Conclusion

Les stubs et placeholders sont identifiés et traçables.
Ce document constitue une base opérationnelle pour planifier des corrections incrémentales vérifiables.
