# Audit approfondi du module `ipc` (Exo-OS)

Date: 2026-03-22
Périmètre: `kernel/src/ipc/**`
Objectif: base de refonte/correction de la couche IPC

---

## 1) Résumé exécutif

Le module `ipc` est la couche 2a de communication inter-processus.
Il fournit rings, channels, endpoints, SHM, RPC, messages, stats et primitives sync.
Il se situe au carrefour de `scheduler`, `memory`, `security`.
Il impacte directement latence et débit du système.

Forces:
- découpage interne riche et explicite.
- API globale exposée via `ipc/mod.rs`.
- présence de fast path ASM pour petite messagerie.
- structures SHM dédiées avec hooks mapping.

Fragilités:
- nombreuses interfaces pouvant dériver (ABI interne).
- certaines zones encore stub/proto.
- forte sensibilité lock ordering IPC<Scheduler<Memory<FS.
- dette potentielle sur fallback mapping SHM.

---

## 2) Positionnement et invariants

`ipc` doit rester indépendant de `fs` et `process` côté import direct.
`ipc` dépend de `memory` pour SHM/futex/maps.
`ipc` dépend de `scheduler` pour block/wake hooks.
`ipc` dépend de `security` pour contrôle d’accès/capacités.
`ipc` doit préserver ordering des messages selon types/channels.
`ipc` doit garantir robustesse sous contention.
`ipc` doit éviter allocations inutiles en fast-path.
`ipc` doit exposer un chemin fallback en absence de hooks.

Invariants clés:
- `ipc_init()` appelé une seule fois.
- scheduler hooks installés avant wait bloquants.
- hooks VMM installés avant mappings SHM réels.
- `futex` IPC appuie sur table unique mémoire.
- IDs monotones et non ambigus.

## 5) Synchronisation, lock ordering, atomiques

Patterns observés:
- `SpinLock` (hooks, tables SHM, registry state).
- atomiques dans stats et états de canaux/rings.
- structures lock-free sur SPSC/MPMC selon implémentations.

Règle d’ordre:
- locks IPC doivent être niveau le plus bas du noyau (avant scheduler/memory/fs).

Points sensibles:
- `sched_hooks` : pont vers blocage/réveil réel.
- `mapping` SHM : cohérence hooks VMM.
- `futex` : alignement avec table mémoire globale.

---

## 6) `&str` et empreinte texte

Usage `&str` observé dans:
- APIs naming/diagnostic du routeur/rpc/stats.
- helpers d’affichage/debug.

Règle refonte:
- préserver logs utiles sans polluer fast-path.
- éviter formatting coûteux en section critique.

---

## 7) TODO/stub/placeholder relevés

Point saillant:
- `shared_memory/mapping.rs` contient un mode stub (virt=phys) dans certains chemins.

Impacts:
- isolation d’espace partiellement simulée en mode fallback.
- comportement acceptable en dev mais fragile en prod multi-processus.

## 9) Conclusion

Le module `ipc` est riche et globalement bien structuré.
Les priorités de correction: hooks scheduler/VMM, chemins SHM fallback, stabilité fast-path.
La refonte doit conserver l’ordre de verrouillage et la compatibilité API noyau.
Ce document sert de baseline de non-régression et de plan d’action.
