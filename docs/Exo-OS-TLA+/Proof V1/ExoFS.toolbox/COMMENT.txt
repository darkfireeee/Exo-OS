### 7.2 Properties: The TLA+ Translation

Here is how we formally write your invariants so TLC doesn't crash:

#### **S9 — Blake3 verified BEFORE decompression**
Instead of `BEFORE`, we enforce that a read operation must transition strictly through specific stages. If a process is in the `DECOMPRESSED` state, it *must* have transitioned from the `CHECKSUMMED` state.
```tla
\* We use a variable ReadOpStage[proc] \in {"IDLE", "READ_DISK", "CHECKSUMMED", "DECOMPRESSED"}
S9_ChecksumBeforeDecompression ==
    [][ \A p \in PROCESSES : 
        (ReadOpStage'[p] = "DECOMPRESSED") => (ReadOpStage[p] = "CHECKSUMMED")
    ]_vars
```

#### **S29 & S30 — ZERO_BLOB Protection**
In TLA+, we can represent `INFINITY` as a special integer (like `-1`) or a string.

S29_ZeroBlobRefcountProtected ==

S30_ZeroBlobOnlyForFullPages ==

#### **S31 — VFS_Tree after NVMe FLUSH**
This is an **Action Property**. You cannot put primed variables (`VfsTree'`) inside a standard `[]` invariant without wrapping the whole thing in the `vars` context. 
The rule: *If a flush is pending, the VFS tree cannot change in the next step.*
```tla
S31_NoPhantomVfsPointers ==
    [][ NvmeFlushPending => (VfsTree' = VfsTree) ]_vars
```
