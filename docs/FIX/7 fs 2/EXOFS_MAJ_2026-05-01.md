# ExoFS MAJ 2026-05-01

## Portee

Cette MAJ consolide les modules ExoFS deja pousses dans les vagues precedentes et verrouille:

- `gc + recovery`
- `audit + observability + numa`
- `relation + quota`
- les chemins transverses `cache + objets + path`
- `compression + dedup`
- `snapshot + export`
- `posix_bridge + VFS + virtio`
- l'export public des appels syscall FS/ExoFS

## Correctifs techniques de cette passe

### 1. Nettoyage des tests et des resets

- suppression des 2 derniers `expect()` textuels restants dans `kernel/src/fs/exofs/tests/`
- nettoyage des imports inutilises dans `test_standard.rs`
- correction du bloc de test `#[allow(deprecated)]` pour eviter le warning parasite
- resets de tests renforces avec `BLOB_CACHE.flush_all_force()` dans:
  - `kernel/src/fs/exofs/posix_bridge/vfs_compat.rs`
  - `kernel/src/syscall/fs_bridge.rs`

### 2. Nettoyage du cache orchestrateur

- consommation explicite du resultat de `BLOB_CACHE.flush_all()` dans:
  - `kernel/src/fs/exofs/cache/mod.rs::shutdown()`
  - `kernel/src/fs/exofs/cache/mod.rs::reclaim_with_report()`

Objectif: run de validation plus propre, sans masquer une erreur par un `must_use` ignore par accident.

### 3. Export syscall FS / ExoFS

`kernel/src/syscall/mod.rs` re-exporte maintenant explicitement:

- les points d'entree FS POSIX bridge:
  - `fs_open`, `fs_openat`, `fs_read`, `fs_write`, `fs_close`
  - `fs_stat`, `fs_lstat`, `fs_fstat`, `fs_lseek`
  - `fs_mkdir`, `fs_rmdir`, `fs_unlink`
  - `fs_getdents64`, `fs_readlink`, `fs_readlinkat`
  - `fs_symlink`, `fs_symlinkat`
  - `fs_dup`, `fs_dup2`, `fs_fcntl`
  - `fs_bridge_init`, `is_fs_ready`, `FsBridgeError`, `fs_bridge_result`

- les handlers ExoFS natifs:
  - `dispatch_exofs_syscall`, `dispatch_exofs_syscall_counted`
  - `sys_exofs_path_resolve`
  - `sys_exofs_object_open/read/write/create/delete/stat/set_meta`
  - `sys_exofs_get_content_hash`
  - `sys_exofs_snapshot_create/list/mount`
  - `sys_exofs_relation_create/query`
  - `sys_exofs_gc_trigger`
  - `sys_exofs_quota_query`
  - `sys_exofs_export_object`, `sys_exofs_import_object`
  - `sys_exofs_epoch_commit`
  - `sys_exofs_open_by_path`
  - `sys_exofs_readdir`
  - `ExofsSyscallArgs`

Un test de smoke export a aussi ete ajoute dans `kernel/src/syscall/mod.rs`.

## Lecture realiste de l'etat

Sur le perimetre effectivement execute ici, ExoFS est maintenant dans une zone de confiance tres haute sur:

- le coeur stockage/objets/path
- les syscalls ExoFS natifs

Le point cle n'est plus seulement "ca compile", mais "les chemins standard + stress + sous-modules critiques passent ensemble".

## Passe finale - recovery bloc-safe et durabilite reelle

Cette derniere passe a cible des ecarts qui ne ressortaient pas forcement comme des tests rouges, mais qui empechaient une lecture honnete a 100% fonctionnel sur un vrai peripherique bloc.

### 1. Recovery: I/O partielles recablees proprement

Les modules suivants utilisent maintenant `kernel/src/fs/exofs/recovery/block_io.rs` pour rester coherents avec le contrat `BlockDevice` (lecture/ecriture de blocs entiers seulement):

- `kernel/src/fs/exofs/recovery/fsck_phase2.rs`
- `kernel/src/fs/exofs/recovery/fsck_phase4.rs`
- `kernel/src/fs/exofs/recovery/slot_recovery.rs`
- `kernel/src/fs/exofs/recovery/epoch_replay.rs`

Impact direct:

- lecture correcte des headers 64/96/128 octets sur devices 4K stricts
- lecture correcte des payloads multi-blocs au recovery
- ecriture correcte des slots et journaux packes sans violer le contrat bloc

### 2. Phase 2 fsck: vrai signal reachable/orphan

Corrections appliquees dans `kernel/src/fs/exofs/recovery/fsck_phase2.rs`:

- ajout de `alloc_entries` dans `Phase2Report`
- suppression du faux double comptage `blobs_walked`
- correction de la taille on-disk des `AllocEntry` lues depuis la table d'allocation
- resolution de reachability via chaine `parent_id -> racine`
- blobs sans chemin vers une racine marques comme orphelins

Resultat:

- `fsck_phase4` n'est plus alimente par un iterateur vide
- la detection d'orphelins ne depend plus d'un faux compteur toujours non nul

### 3. Phase 4 fsck: recuperation multi-bloc reelle

Corrections appliquees dans `kernel/src/fs/exofs/recovery/fsck_phase4.rs`:

- suppression du stub `alloc_entries_iter()`
- verification HASH-02 sur blobs multi-blocs via `read_bytes`
- persistance `lost+found` recablee de facon bloc-safe

### 4. Journal d'epoch: format 64B verrouille

Correction structurelle dans `kernel/src/fs/exofs/recovery/epoch_replay.rs`:

- `EpochJournalHeaderDisk` refixe a `64` octets reels
- `const assert` reactive
- lecture d'enregistrements packes a offset byte-precis dans un meme bloc

### 5. Flush/Barrier: durabilite branchee au disque global

Corrections appliquees dans:

- `kernel/src/fs/exofs/storage/virtio_adapter.rs`
- `kernel/src/fs/exofs/mod.rs`

Changements:

- `VirtioBlockAdapter::flush()` delegue maintenant au backend bloc
- ajout de `flush_global_disk()`
- enregistrement du hook `epoch_barriers::register_nvme_flush_fn(...)` pendant `exofs_init()`

Effet:

- le protocole de barrieres epoch n'est plus laisse sur le chemin stub par defaut apres init ExoFS

### 6. Tests ajoutes pour les cas realistes

Nouveaux tests cibles:

- `fs::exofs::recovery::fsck_phase2::tests::test_phase2_detects_parentless_reachability_and_orphans`
- `fs::exofs::recovery::fsck_phase4::tests::test_phase4_recovers_multiblock_orphan_on_strict_device`
- `fs::exofs::recovery::slot_recovery::tests::test_write_slot_roundtrip_on_strict_block_device`
- `fs::exofs::recovery::epoch_replay::tests::test_validate_journal_reads_packed_records_on_strict_block_device`
