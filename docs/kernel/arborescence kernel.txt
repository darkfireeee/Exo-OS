## 📋 Changelog v5 — 4 corrections intégrées

### ✅ Correction 1 — `signal/` déplacé de `scheduler/` vers `process/`
Les signaux POSIX (SIGCHLD, SIGTERM, sigaltstack…) appartiennent au cycle de vie
processus. Seule `delivery.rs` touche le scheduler (wakeup au retour kernel).
`scheduler/` ne contient plus `signal/`.

### ✅ Correction 2 — `capability/` déplacé vers `security/` + bridge dans `ipc/`
- `security/capability/` : modèle formel complet, périmètre de preuve minimal
- `ipc/capability_bridge/` : shim zéro-logique qui délègue à `security/capability/`
- `fs/` appelle `security/capability/` directement, sans passer par `ipc/`

### ✅ Correction 3 — Garantie formelle r15 dans `scheduler/asm/switch_asm.s`
Le commentaire vague est remplacé par une garantie explicite : r15 est push/pop
**avant tout appel C**, ce qui couvre l'usage de r15 dans `ext4plus/inode/ops.rs`.
Le commentaire correspondant dans `ops.rs` est mis à jour en conséquence.

### ✅ Correction 4 — Modules IA réintégrés
- `memory/physical/allocator/ai_hints.rs` : lookup table statique NUMA, zéro inférence runtime
- `scheduler/policies/ai_guided.rs` : politique guidée par profil de charge, fallback CFS si hints absents

### ✅ Corrections antérieures (v4 → v5 conservées)
- `pku.rs` : "Intel MPX" → "Protection Keys for Userspace (Intel)"
- `aes_gcm.rs` : "fallback HW AES-NI" → "accélération HW AES-NI"
- Séparation sémantique FPU clarifiée : `arch/` = ASM brut, `scheduler/fpu/` = logique état TCB
- DMA conservé dans `memory/` — dépendance vers `process/` résolue via trait `WakeupHandler`
  enregistré au boot dans `process/state/wakeup.rs`
